How should the daily report backend be structured?

Three approaches — pick the one that fits best

A

Plug into existing worker.ts + queue-service.ts ✅ Recommended

Add a daily-report BullMQ queue to the existing queue infrastructure. The cron fires inside worker.ts at 11:59 PM and enqueues one job per enabled user. A new daily-report-worker-service.ts processes them and sends via Brevo. The daily_report_enabled boolean column is added to the users table.

Pros

  • Reuses existing Redis + BullMQ infra
  • One process to deploy and monitor
  • Consistent with campaign/sequence pattern already in codebase
  • Retry logic already configured

Cons

  • worker.ts gets one more responsibility
B

Separate standalone Node.js process

A completely separate /src directory (as in your code snippet) that runs independently — its own Redis connection, its own cron, its own worker. Deployed as a separate container or process alongside the main API.

Pros

  • Fully isolated — can be deployed/scaled independently
  • Matches the code you already wrote

Cons

  • Duplicates Redis connection setup
  • Needs its own DB connection to query users + conversations
  • More infra to maintain and deploy
  • Two processes to monitor
C

Simple cron-only inside the API (no queue)

A cron job that runs at 11:59 PM inside app.ts, loops through all enabled users, generates + sends each email inline — no BullMQ involved.

Pros

  • Simplest possible implementation
  • No queue infrastructure needed

Cons

  • No retry on email failure
  • Blocks if many users — will slow down as you scale
  • Not consistent with existing patterns
