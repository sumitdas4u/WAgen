3 approaches to keyword routing

All three keep backward compatibility with existing flows

▶ Flow Start
Route: "Greeting"
🔑 hello
🔑 hi
Route: "Orders"
🔑 order
Default (no match)

A — Routes with nested triggers ✦ recommended

Add routes[] to FlowStartData. Each route has a label, its own triggers list, and one output handle. A separate "Default" handle catches unmatched messages.

Pros

  • Clean data model
  • Clear UI grouping
  • Scales to many routes

Cons

  • Bigger refactor — engine must track which trigger matched
▶ Flow Start

Greeting 🔑 hi
Orders 🔑 order
Default ↓

B — Route tag on each trigger

Keep flat triggers[], add a routeId field per trigger, plus a separate routes[] name registry. Less structural change.

Pros

  • Smaller change to existing data shape

Cons

  • Harder to see groupings at a glance
  • Two arrays to keep in sync

C — Save matched keyword as variable

FlowStart saves the matched keyword into {{matched_trigger}}. Users chain Condition nodes to branch. No UI change to Flow Start.

Pros

  • Minimal code change
  • Reuses existing Condition node

Cons

  • Verbose — need N condition nodes for N routes
  • Not visual from start node
